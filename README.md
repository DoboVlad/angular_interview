# Blue Eclipse — Senior Angular Interview Quiz

A premium, fully self-contained Angular 22 single-page app for senior engineers
(3–8+ years) practising for technical interviews. 110 reasoning-first questions
across Angular, JavaScript/TypeScript and the browser platform — graded,
adaptive, and explained, with optional AI deep-dives via `claude-fable-5`.

Built on the **Blue Eclipse** palette, Angular **Signals** throughout, **zoneless**
change detection, **Signal Forms**, `resource()`/`rxResource()` for async, and
lazy-loaded routes with the View Transitions API.

> ⚠️ **Bleeding-edge APIs.** This app targets Angular 22 and uses experimental
> surfaces (zoneless CD, `resource()`, signal queries, deferrable views). Run
> `npm install` against your installed Angular 22 toolchain; if an experimental
> import has been renamed in your exact build, the fix is a one-line import
> change. Signal Forms is provided as a small local toolkit
> (`src/app/shared/forms/signal-forms.ts`) that mirrors the framework API — swap
> it for the built-in when it stabilises with no change to call sites.

## Quick start

```bash
npm install
npm start            # ng serve  → http://localhost:4200
npm run build:prod   # production build → dist/blue-eclipse-quiz/browser/
npm test             # vitest watch
npm run test:run     # vitest once (CI)
```

The 110-question bank lives in [`src/data/questions.json`](src/data/questions.json)
and is generated by [`scripts/gen-questions.mjs`](scripts/gen-questions.mjs),
which also **validates the composition rules** (category mix, exact difficulty
spread, ≥5 code snippets). Regenerate with:

```bash
node scripts/gen-questions.mjs
```

## Features

| Screen        | Highlights |
|---------------|-----------|
| **Dashboard** | Signal-Form test config (size / focus / timer), `@defer`-ed stats strip, recent history |
| **Test**      | SVG countdown ring, keyboard nav (`1–4`, `Enter`, `→`), immediate feedback, explanation + senior insight reveal, streaming AI deep-dive |
| **Results**   | Score hero + letter grade, pure-SVG donut, per-topic breakdown, missed-question accordion, share-to-clipboard |
| **Browse**    | CDK virtual scroll over all 110 questions, Signal-Form multi-select filters, debounced signal search, slide-in detail |
| **Settings**  | Client-side API key (masked, show/hide), default timer, theme (light/dark/system), reset all data |

**Adaptive nudge:** three consecutive correct answers at level *N* fires a
“Level up!” toast and pulls higher-difficulty questions forward in the queue.

## AI explanations (`claude-fable-5`)

The **Go deeper with AI** button streams a principal-engineer explanation
directly from the browser to the Anthropic Messages API:

- Model `claude-fable-5`, `max_tokens: 1500`, `stream: true`.
- SSE tokens are parsed and appended character-by-character, rendered as
  markdown via `marked`.
- Abortable with a **Stop** button (`AbortController`); error states cover no
  key, auth, rate-limit (with retry-after), and network.
- The API key is entered in **Settings** and stored **only** in this browser’s
  `localStorage`. It is sent straight to Anthropic and never to a server of ours.
  Use a personal key, never a shared/production one.

Because the call is browser-to-API, the request sends the
`anthropic-dangerous-direct-browser-access: true` header and your hosting/CDN
must allow the cross-origin request (see CORS below).

## Deploying to AWS S3 (+ CloudFront)

### 1. Build

```bash
npm run build:prod
# output: dist/blue-eclipse-quiz/browser/
```

### 2. Create + configure the bucket

```bash
aws s3 mb s3://your-bucket-name
aws s3 website s3://your-bucket-name --index-document index.html --error-document index.html
```

The SPA error document is `index.html` so deep links (`/results/:id`) resolve.

**Bucket policy** (public read of objects — for a public study site):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::your-bucket-name/*"
    }
  ]
}
```

(Disable “Block all public access” on the bucket for the website endpoint, or
keep it private and serve via CloudFront + OAC — recommended for production.)

### 3. CORS

CORS on the bucket is only needed if assets are fetched cross-origin; the
**Anthropic API call is governed by Anthropic’s own CORS**, not your bucket.
For completeness, a permissive bucket CORS config:

```json
[
  {
    "AllowedOrigins": ["*"],
    "AllowedMethods": ["GET", "POST"],
    "AllowedHeaders": ["*"],
    "MaxAgeSeconds": 3000
  }
]
```

Tighten `AllowedOrigins` to your domain in production.

### 4. Deploy script

```bash
chmod +x deploy.sh
S3_BUCKET=your-bucket-name ./deploy.sh
# with CloudFront invalidation:
S3_BUCKET=your-bucket-name CLOUDFRONT_DIST_ID=E123ABC AWS_REGION=eu-west-1 ./deploy.sh
```

`deploy.sh` builds, syncs hashed assets with a one-year cache, uploads
`index.html` with `no-cache`, and invalidates CloudFront when `CLOUDFRONT_DIST_ID`
is set.

### 5. CloudFront (HTTPS + custom domain)

1. Create a distribution with the S3 bucket as origin (use **Origin Access
   Control** if the bucket is private).
2. Set **Default root object** to `index.html`.
3. Add a **custom error response**: HTTP 403/404 → `/index.html`, response code
   200 (so client-side routes work).
4. Attach an **ACM certificate** (in `us-east-1`) for your domain and add the
   CNAME; point Route 53 / DNS at the distribution.

### 6. Automated deployment (GitHub Actions)

[`.github/workflows/deploy.yml`](.github/workflows/deploy.yml) builds + tests on
every push/PR and **deploys to S3 on push to `main`**. It authenticates to AWS
with **OIDC** (a short-lived assumed role — no long-lived keys in the repo).

**One-time AWS setup:**

1. Add GitHub as an OIDC identity provider in IAM:
   - Provider URL: `https://token.actions.githubusercontent.com`
   - Audience: `sts.amazonaws.com`
2. Create an IAM role with a trust policy scoped to this repo, e.g.:
   ```json
   {
     "Effect": "Allow",
     "Principal": { "Federated": "arn:aws:iam::<ACCOUNT_ID>:oidc-provider/token.actions.githubusercontent.com" },
     "Action": "sts:AssumeRoleWithWebIdentity",
     "Condition": {
       "StringEquals": { "token.actions.githubusercontent.com:aud": "sts.amazonaws.com" },
       "StringLike": { "token.actions.githubusercontent.com:sub": "repo:DoboVlad/angular_interview:*" }
     }
   }
   ```
3. Give the role permissions for `s3:PutObject`, `s3:DeleteObject`, `s3:ListBucket`
   on the bucket, plus `cloudfront:CreateInvalidation` if you use CloudFront.

**Configure the repo** (Settings → Secrets and variables → Actions):

| Type     | Name                  | Example |
|----------|-----------------------|---------|
| Secret   | `AWS_ROLE_ARN`        | `arn:aws:iam::123456789012:role/github-deploy` |
| Variable | `AWS_REGION`          | `eu-west-1` |
| Variable | `S3_BUCKET`           | `your-bucket-name` |
| Variable | `CLOUDFRONT_DIST_ID`  | `E123ABC` (optional — leave unset to skip invalidation) |

That's it: push to `main` and the site redeploys. The `production`
[environment](https://docs.github.com/actions/deployment/targeting-different-environments)
in the workflow lets you add a required-reviewer gate if you want manual approval
before deploys.

> Prefer access keys over OIDC? Swap the `configure-aws-credentials` step inputs
> for `aws-access-key-id`/`aws-secret-access-key` secrets — but OIDC is safer.

## Quality gates

- TypeScript strict mode — no `any`, no unexplained non-null assertions.
- All components `OnPush`; app runs **zoneless**.
- No direct DOM mutation for state — signal-driven bindings only.
- All routes lazy-loaded; preloaded with `PreloadAllModules`.
- No leaks: `DestroyRef`, `takeUntilDestroyed()`, `AbortController` cleanup.
- Accessibility: ARIA roles/labels, keyboard-navigable, skip link, reduced-motion.
- No `console.*` in production paths (dev-guarded in `StorageService`).
